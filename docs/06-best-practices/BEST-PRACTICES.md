# Best Practices: การเขียนวิทยานิพนธ์ด้วย MCP Thai Thesis

> แนวปฏิบัติที่ดีที่สุดจากประสบการณ์การเขียนวิทยานิพนธ์จริงด้วยระบบ MCP Thai Thesis

---

## 1. หลักการพื้นฐาน

### 1.1 ลำดับการใช้ Tools ที่แนะนำ

**สิ่งที่ได้ผลจริง:**
```
เขียน Draft → De-AI Scan → แก้ไขตาม De-AI → Review → Rewrite (ถ้าต้องการ) → Score
```

**สิ่งที่ควรทำ:**
1. เขียน draft ก่อน (ใช้ master prompt หรือ /thesis-chapter)
2. รัน De-AI scan เสมอ ก่อน review
3. แก้ไข banned patterns และปรับ CV ก่อนเรียก advisor review
4. ใช้ /thesis-score เป็น checkpoints ระหว่างทาง

**สิ่งที่ไม่ควรทำ:**
- ข้าม De-AI scan ก่อนส่ง advisor
- เขียนทั้งเล่มแล้วค่อย scan ทีเดียว — แก้ยาก
- ใช้ rewrite mode ตั้งแต่แรก — เสียเนื้อหาที่ดี

### 1.2 การโหลด Skills

| สถานการณ์ | Skill ที่ควรโหลด |
|-----------|------------------|
| เขียนบทใหม่ | `thai-thesis-writing` |
| ตรวจ/ลบ AI patterns | `thai-de-ai` |
| จัดการอ้างอิง | `thai-citation-manager` |
| สร้าง ISO เอกสาร | `iso29110-docs` |
| ตรวจก่อนส่ง | `thesis-quality-gate` |

## 2. Master Prompt

### 2.1 โครงสร้าง Master Prompt ที่ใช้จริง (12 Sections)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Master Prompt Structure                           │
├─────────────────────────────────────────────────────────────────────┤
│ 1. Role: นิยามตัวตน AI (ผู้เชี่ยวชาญด้าน...)                    │
│ 2. ข้อมูลนักศึกษา/สถาบัน (ชื่อ, รหัส, อาจารย์)                   │
│ 3. ชื่อวิทยานิพนธ์ (ไทย + อังกฤษ + คำสำคัญ)                       │
│ 4. กฎการเขียน (ภาษา, APA, De-AI)                                 │
│ 5. โครงสร้างวิทยานิพนธ์ (7 บท พร้อม subsections)                  │
│ 6. ข้อมูลโครงการ (technical specs, architecture)                   │
│ 7. ข้อมูลการวิจัย (experiments, data, results)                     │
│ 8. แผนการทดลอง (methodology, sample size)                          │
│ 9. ISO 29110 docs (file references)                                │
│ 10. Reference files (paths to actual files)                       │
│ 11. เคล็ดลับการเขียนแต่ละบท                                       │
│ 12. ข้อมูลที่มี vs [TBD] / [CITATION NEEDED]                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 หลักการเขียน Master Prompt

1. **รวมทุกอย่างไว้ใน prompt เดียว** — AI ไม่ต้องอ่านไฟล์เพิ่ม
2. **ใส่ข้อมูลจริงให้มากที่สุด** — ยิ่งละเอียด ยิ่งดี
3. **ระบุ [TBD] ชัดเจน** — บอก AI ว่าอะไรยังไม่มี
4. **ใส่ file references** — ให้ AI รู้ว่าข้อมูลอยู่ที่ไหน

### 2.3 ตัวอย่าง Section 4 (กฎการเขียน)

```markdown
## กฎการเขียน

### ภาษาและรูปแบบ
- ภาษาไทยวิชาการ ตามหลัก ราชบัณฑิตยสถาน
- สรรพนาม: "ผู้วิจัย" (ไม่ใช้ "ผม", "ฉัน", "เรา")
- เว้นวรรคหลังเครื่องหมายวรรคตอน
- ตัวเลข: อารบิก (1, 2, 3) ยกเว้น 1-10 ต้นประโยค

### การอ้างอิง
- รูปแบบ: APA นาม-ปี (ไม่ใช้ตัวเลข)
- ระบบปี: พ.ศ. (เช่น 2566)
- ห้ามปลอมแปลง — ใช้ [CITATION NEEDED]

### กฎ De-AI
- ค่า CV >= 30% (Burstiness)
- ห้าม banned patterns:
  - นอกจากนี้ (เกิน 3 ครั้ง/บท)
  - ก้าวสำคัญ, พลิกโฉม, ล้ำสมัย
  - ประโยคว่างเปล่าปิดท้าย
- ผสมประโยคสั้น-กลาง-ยาว
```

## 3. De-AI Best Practices

### 3.1 ค่า CV (Coefficient of Variation)

**สิ่งที่ได้ผลจริง:**
- CV >= 30% = human-like ✓
- CV < 30% = likely AI ✗

**วิธีเพิ่ม CV:**
1. ผสมประโยคสั้น (5-10 คำ) กับยาว (30+ คำ)
2. เปลี่ยนโครงสร้างประโยค: กรรม-กริยา-กรรม → กรรม-กริยา-กรรม → เริ่มต้นด้วย adverb
3. ใช้ "ผู้วิจัย" สลับกับ passive voice

### 3.2 Banned Patterns

**คำเชื่อมที่ควรหลีกเลี่ยง:**
| คำ | แทนที่ด้วย |
|----|----------|
| นอกจากนี้ | อีกประการหนึ่ง, นอกจากนี้ (ใช้ได้แต่ไม่เกิน 3 ครั้ง/บท) |
| ยิ่งไปกว่านั้น | ยิ่งไปกว่านี้, นอกจากนี้ |
| ทั้งนี้ | โดยเฉพาะอย่างยิ่ง, อีกประการ |

**คำโฆษณาที่ห้ามใช้:**
- ก้าวสำคัญ, พลิกโฉม, ล้ำสมัย, เปลี่ยนแปลงโลก
- ยกระดับ, พัฒนาก้าวกระโดด, แตะขีดจำกัด
- ปฏิวัติ, นวัตกรรมล้ำ, เหนือกว่า

### 3.3 Soul Injection Techniques

**วิธีทำให้เนื้อหาดูเป็นมนุษย์มากขึ้น:**

1. **เพิ่ม personal anecdote:**
   ```markdown
   # แทนที่:
   "การปรับแต่ง Windows ช่วยลด latency ได้"
   
   # ใช้:
   "จากประสบการณ์การเล่น Valorant ของผู้วิจัยเอง พบว่า..."
   ```

2. **เพิ่ม hedge language:**
   ```markdown
   # แทนที่:
   "ผลการวิจัยแสดงว่า..."
   
   # ใช้:
   "ผลการวิจัยมีแนวโน้มที่จะแสดงว่า..."
   ```

3. **เพิ่ม limitation acknowledgment:**
   ```markdown
   "อย่างไรก็ตาม การศึกษานี้มีข้อจำกัดด้าน..."
   ```

## 4. Citation Best Practices

### 4.1 APA นาม-ปี

**รูปแบบที่ถูกต้อง:**
- แบบประโยค: `ผู้แต่ง (2566) กล่าวว่า...`
- แบบวงเล็บ: `(ผู้แต่ง, 2566)`
- หลายคน: `(ผู้แต่งที่ 1 และผู้แต่งที่ 2, 2566)`

### 4.2 ระบบปี

| ประเภทผู้แต่ง | ระบบแนะนำ |
|--------------|----------|
| ไทย | พ.ศ. (2566) |
| อังกฤษ | พ.ศ. หรือ ค.ศ. (กำหนดเอง แต่ต้องตรงกันทั้งเอกสาร) |

**ข้อควรระวัง:** ห้ามผสม พ.ศ. และ ค.ศ. ในเอกสารเดียวกันโดยไม่ได้ตั้งใจ

### 4.3 การใช้ [CITATION NEEDED]

- ใช้เมื่อไม่มีข้อมูลจริง — **ห้ามสมมติ**
- ทำเป็น list และค้นหาทีหลัง
- ตรวจสอบด้วย `thai_citation action=audit`

## 5. Quality Gate Best Practices

### 5.1 8-Gate Framework

```
Gate 1: โครงสร้าง      → มีครบทุกบท (1-7)
Gate 2: เนื้อหา        → ลึกซึ้ง, มีเหตุผล
Gate 3: การอ้างอิง    → APA ถูกต้อง
Gate 4: ภาษาไทย       → ราชบัณฑิตยสถาน
Gate 5: De-AI         → CV >= 30, 0 banned
Gate 6: ความสอดคล้อง → ตัวเลข, ศัพท์ตรงกัน
Gate 7: รูปแบบ        → ตรง template
Gate 8: ISO 29110     → ครบ 10 เอกสาร
```

### 5.2 จังหวะการตรวจ

| ช่วง | Gate ที่ควรตรวจ |
|------|-----------------|
| หลังเขียนแต่ละบท | Gate 1, 4, 5 |
| หลังเขียนบท 2 | Gate 3 (เริ่มหาอ้างอิง) |
| หลังเขียนบท 6 | Gate 6 |
| ก่อนส่ง | Gate 7, 8 + Full Audit |

## 6. ISO 29110 Best Practices

### 6.1 เอกสารที่ต้องมี (10 ฉบับ)

| เอกสาร | เวลาสร้าง | บทที่เกี่ยวข้อง |
|--------|----------|----------------|
| Project Plan | เริ่มต้น | บท 1-3 |
| SRS | หลัง บท 2 | บท 3 |
| SDD | หลัง บท 3 | บท 4 |
| Test Plan | หลัง บท 4 | บท 5 |
| Test Record | หลัง บท 6 | บท 6 |
| Traceability | หลัง Test Record | บท 6 |
| Change Request | ถ้ามีการเปลี่ยนแปลง | - |
| Progress Report | ตามความต้องการ | - |
| Config Plan | ถ้ามี | บท 5 |
| User Manual | ก่อนส่ง | ภาคผนวก |

### 6.2 ข้อแนะนำ

- สร้างเป็น bilingual (ไทย-อังกฤษ) เสมอ
- ใช้ `iso_document` tool แล้วแก้ไขเอง
- อ้างอิงจากวิทยานิพนธ์ให้ตรงกัน

## 7. สิ่งที่ได้ผล vs ไม่ได้ผล

### 7.1 สิ่งที่ได้ผล

| สิ่งที่ทำ | ผลลัพธ์ |
|---------|--------|
| เขียนทีละบท แล้ว De-AI ทันที | แก้ง่าย, ไม่สะสม |
| ใช้ master prompt ที่ละเอียด | ได้เนื้อหาครบถ้วน |
| ตรวจ CV หลังเขียนทุกครั้ง | ไม่ต้อง rewrite ทั้งบท |
| ค้นอ้างอิงตั้งแต่ บท 2 | bibliography ครบก่อนส่ง |
| รัน consistency check ก่อนส่ง | ไม่มีตัวเลขผิด |

### 7.2 สิ่งที่ไม่ได้ผล

| สิ่งที่ทำ | ปัญหา |
|---------|--------|
| เขียนทั้งเล่มก่อนค่อย De-AI | แก้ยากมาก, ใช้เวลานาน |
| ใช้ rewrite mode ตั้งแต่แรก | เสียเนื้อหาที่ดี, AI อาจเขียนผิด |
| ไม่ใช้ [CITATION NEEDED] | สมมติข้อมูล, ผิดจริง |
| ข้าม consistency check | ตัวเลขไม่ตรงกัน |
| เขียน ISO ทีหลัง | ไม่สอดคล้องกับเนื้อหา |

## 9. Checklist ก่อนส่งวิทยานิพนธ์

- [ ] โครงสร้าง: ครบ 7 บท
- [ ] เนื้อหา: ครบตามขอบเขต
- [ ] อ้างอิง: ไม่มี [CITATION NEEDED] ค้างอยู่
- [ ] De-AI: CV >= 30, 0 banned patterns
- [ ] Bibliography: เรียงตามตัวอักษร, รูปแบบถูกต้อง
- [ ] รูปแบบ: ตรง template มหาวิทยาลัย
- [ ] ISO: ครบ 10 เอกสาร
- [ ] ความสอดคล้อง: ผ่าน consistency check
- [ ] คะแนน: ได้ >= 75/100
- [ ] Gates: ผ่าน 8 gates
